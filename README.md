# ResearchFactor (RF-Protocol): Core Data Engine

## Project Context
This is a scientific data management system. Its goal is to create an immutable, step-by-step record of a research process. This version focuses on **Research Definition** and **Execution** only (Review and Funding excluded).

## 1. Domain Entities (Core Hierarchy)

### A. Research Project
The top-level container for a scientific goal.
- `id`: UUID
- `title`: String
- `description`: Markdown
- `lines`: ResearchLine[] (Ordered)

### B. ResearchLine (Phase)
A major block of work representing a specific experimental phase.
- `id`: UUID
- `sequenceOrder`: Integer (Determines execution flow)
- `title`: String (e.g., "In-vitro testing")
- `status`: `LOCKED | ACTIVE | COMPLETED`
- `objectives`: Objective[] (Expected outcomes for this line)
- `tasks`: ResearchTask[] (Actual work performed)

### C. Objective (The "Target")
Pre-defined success criteria. Immutable once the ResearchLine starts.
- `id`: UUID
- `description`: String (e.g., "Glucose reduction > 10%")
- `status`: `PENDING | FULFILLED | FAILED` (Set manually or by script)

### D. ResearchTask (The "Action")
A granular entry in the lab journal.
- `id`: UUID
- `status`: `DRAFT | SUBMITTED`
- `logEntries`: List<LogEntry> (Real-time notes with immutable timestamps)
- `artifacts`: Artifact[] (Raw data/evidence)

### E. Artifact (The "Evidence")
- `id`: UUID
- `type`: `RAW_DATA | PHOTO | CODE | CONFIG`
- `storageUrl`: String
- `sha256`: String (File integrity)
- `metadata`: Map<String, String> (Device IDs, environment specs, etc.)

## 2. Technical Requirements (Angular 21)

### State Management (Signals Only)
- Use `Signal` for all UI-bound data.
- Avoid `BehaviorSubject` and `subscriptions`.
- Implement `computed` values for progress tracking (e.g., `completedTasksPercent`).

### Immutability Rules
1. **ResearchLine Locked:** Once a line has `status: COMPLETED`, no more `ResearchTask` can be added or modified.
2. **Task Submission:** Once a `ResearchTask` is `SUBMITTED`, all its `logEntries` and `artifacts` become `ReadOnly`.
3. **Timestamp Integrity:** `LogEntry` timestamps must be generated by the backend; frontend displays them in local time.

- **Framework:** Angular 19+ (Signal-based) using Standalone Components.
- **State Management:** Angular Signals + RxJS for Event Streams.
- **Styling:** Tailwind CSS + Headless UI components.
- **Communication:** REST API (Backend: Java/Spring Boot), WebSocket for Live Lab Logs.
- **Security:** JWT + Role-Based Access Control (RBAC): `RESEARCHER`, `LEAD_PHD`, `SHADOW_PHD`, `DONOR`.

## 3. Workflow for Developer Agent

### Flow 1: Construction (Setup)
- Create `Research` -> Add `ResearchLine` -> Define `Objectives`.
- Define set of `ResearchTask` for `ResearchLine`
- All data is editable in this phase.

### Flow 2: Laboratory Work (Execution)
- Set `ResearchLine` to `ACTIVE`.
- Add `LogEntries` to `ResearchTask` as the work progresses.
- Upload `Artifacts` and link them to the Task.
- Final Action: `SUBMIT_TASK`.

### Flow 3: Progress Tracking
- A `ResearchLine` can be marked `COMPLETED` only if all its `Objectives` have a summary and all `Tasks` are `SUBMITTED`.

---

## 4. UI Architecture Guidelines
- **Vertical Pipeline:** Display `ResearchLines` as a vertical progression.
- **Artifact Sidebar:** Context-aware panel that shows files related to the selected `ResearchTask`.
- **Mobile-Friendly Inputs:** Focus on the `LogEntry` input and `Camera Upload` for mobile lab work.
- **No-Delete Policy:** Implement "Archive" or "Correction" logs instead of deleting existing data entries to preserve the audit trail.
